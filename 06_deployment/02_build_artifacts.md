# 06.2 — Build Artifacts: dist/, build/, wheels, .egg-info

> Goal: understand the generated folders that appear after building/installing, and what "publishing" produces. (These are the "ignore" files from lesson 01.1 — now you'll know what they are.)

---

## "Building" a package = turning source into a shippable bundle

Your source is a folder of `.py` files. To *share* it on PyPI so others can `pip install` it, you **build** it into standard distributable files. The build tool (hatchling/setuptools/poetry) reads `pyproject.toml` and produces artifacts in a `dist/` folder.

```bash
python -m build        # or: uv build, poetry build
# produces:
dist/
├── codehound-0.1.0-py3-none-any.whl    # a WHEEL
└── codehound-0.1.0.tar.gz              # an SDIST
```

---

## The two artifact types

### Wheel (`.whl`) — the pre-built package
A **wheel** is a ready-to-install zip of your package. `pip install` prefers wheels because there's nothing to compile — just unzip into `site-packages`. Fast.
- Filename decoded: `codehound-0.1.0-py3-none-any.whl` = name `codehound`, version `0.1.0`, works on `py3` (any Python 3), `none`/`any` (no special platform). 
- **Your huggingface_hub fix shipped inside the `v1.17.0` wheel** — that's the literal file people download.

### Sdist (`.tar.gz`) — the source distribution
A compressed archive of the *source* (the raw files). pip can build a wheel from it if no wheel exists. The fallback format.

---

## The generated folders you should IGNORE

After building or installing, these appear. They're **all generated** and gitignored — you never edit or commit them:

| Folder/file | What it is | Why ignore |
|-------------|-----------|------------|
| `dist/` | the built wheels/sdists | regenerated by `build` |
| `build/` | intermediate build scratch space | temporary |
| `*.egg-info/` | packaging metadata (name, version, deps) | auto-generated on install |
| `*.egg-link` | editable-install pointer | local only |
| `__pycache__/`, `*.pyc` | Python's compiled bytecode cache | auto-recreated |
| `.pytest_cache/`, `.mypy_cache/`, `.ruff_cache/` | tool caches | auto-recreated |

**The rule from lesson 01.1, now fully explained:** these all exist because tools *generated* them. Deleting them is harmless — they come back. That's exactly why they're in `.gitignore`. When you open a repo and see these, mentally filter them out — they're noise, not content.

---

## What `__pycache__` and `.pyc` actually are

When Python imports a module, it compiles the `.py` to **bytecode** (a lower-level form the interpreter runs faster) and caches it as a `.pyc` file inside `__pycache__/`. Next import, if the `.py` hasn't changed, Python reuses the cached `.pyc` — a speed optimization. It's an implementation detail you never touch.

---

## PyPI — where published packages live

**PyPI** (Python Package Index, pypi.org) is the public registry of Python packages. Maintainers upload wheels there; `pip install X` downloads from there. Publishing = `twine upload dist/*` or `uv publish` or `poetry publish`. (Your codehound *could* be published to PyPI so people could `pip install codehound` — a possible future step.)

The JS equivalent is **npm** (npmjs.com); the container equivalent is **Docker Hub / GHCR**.

## The full lifecycle (tying it together)

```
write source (.py)
   │ build  (reads pyproject.toml)
   ▼
dist/ : wheel + sdist        ← artifacts
   │ publish (twine/uv/poetry)
   ▼
PyPI : pip install codehound works for everyone
   │ a user installs into their .venv
   ▼
their site-packages/ : your code runs in their project
```

Your **huggingface_hub** contribution traveled this whole path — merged → built into the v1.17.0 wheel → published to PyPI → `pip install huggingface_hub==1.17.0` runs your code worldwide. That's the strongest form of "shipped."
